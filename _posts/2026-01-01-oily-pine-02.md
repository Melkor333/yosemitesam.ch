## Intro

I started the Oily Pine project in a try to build all Alpine Linux Packages using [OSH](https://oils.pub/osh.html) as shell.

In the [last article]({{ site.baseurl}}/oily-pine-01) I've shown how to set up a build environment and finished the article with a first run.
This first run resulted in 5327 successfully built packages until the disk filled up, with 1173 failed packages.
Since we want to make OSH behave like Busybox Ash, we only care about the failed packages.

Analyzing 1173 failed package builds manually one by one is a bit much, though. As visible in [this walk-through of a bugfix by Aidan Olsen](https://aolsen.ca/writings/reproducing-dtc-failure), the procedure can take a few hours for a single package.

Luckily, we can assume that many bugs occur in more than just one package. Briefly looking into just a few logs I could confirm that some errors seemed to happen repeatedly.
So instead of jumping straight into deep analysis package by package, we can skim the logs and categorize the errors. In the best case we'll only need to look into a single log-file per category to know how all of them failed. And we can start with the most prevalent bugs first to get the number down as fast as possible.
But even if we only need half a minute to categorize a log-file, it's still almost 10 hours of staring at log files.

But what could be better to churn through so much text and categorize it? A machine?... **AI?!**
I've only used LLMs for output that was verifiable code or translations, but this time the LLM would be "the program" (instead of writing the program)!

Let's try.

## Talking to ChatGPT

So I wrote a small script to send the last 40 lines of each log-file to ChatGPT and try to summarize the problem in *maximum 5 words*. I wrote the script in [YSH](https://oils.pub/ysh.html), because it has a few features which made the below 16 lines of code a bit extra nice.
Since the whole project is about the Oils project, I also like to give you a bit of a showcase/intro into the non-POSIX compliant part of Oils. I really believe in it as a path forward from the minefield that POSIX shell is.

```bash
# simplified categorization-script

#!/usr/bin/env ysh

# *generate a list of log files with errors from the builrepo log files*
var bad_log_files = glob("/path/to/folder/with/failed/logs/*")

for file in (bad_log_files) {
  ... cat ./PROMPT_TEMPLATE.md
      # Ignore all lines from abuild removing packages at the end
    <(... grep -v "Purging" $file | head -n -1
      # the error should be in the last few lines.
      # chatgpt can't handle too much input!
      | tail -40
      ;)
    | chatgpt
    | tr -d u'\n'
    >> $out_file;

  # Add a newline
  echo >> $out_file
}
````

The first line shows us how variable assignments work in YSH:

```bash
var bad_log_files = glob("/path/to/folder/with/failed/logs/*")
```

In bash, it would be `bad_log_files=("/path/to/folder/with/failed/logs/"*)` (note the `*` being outside the quotes). In POSIX the only sane way is to not store it in a variable: `for file in /path/to/folder/with/failed/logs/*`.
At a first glance YSH looks just more verbose. But what actually happens is that after the `=` YSH enters an [expression-mode](https://oils.pub/release/0.36.0/doc/ref/chap-expr-lang.html), which has functions (like `glob`) and is *dynamically typed* like Python or JavaScript. Unquoted globs are not possible in that expression mode, because it would conflict with mathematical multiplication (e.g. `var x = 5*5` is `25`, NOT a list of files starting and ending with a `5`).

The expression mode is probably the biggest difference from POSIX shell to YSH.
At first it seems somewhat random and hard to understand when YSH switches from shell-like "Command Mode" to the "Expression Mode", but it doesn't require much practice to become an intuition. The [Vim Syntax Highlighter](https://github.com/oils-for-unix/oils.vim) can also be a big help, as it changes the color for the expression mode. (A [Pygments](https://pygments.org/) highlighter I could use for this blog is still an open to-do).

Now let's look at the for loop:

```bash
for file in (bad_log_files) {

}
```

In YSH that's how a typical `for` loop looks like. Inside of the parentheses `()` YSH enters expression mode again and expects either a list or dict from the expression.

Another option would be to stay in the command mode and use `for file in @bad_log_files { ... }`. This `@mylist` syntax comes [from Perl](https://www.perl.com/article/on-sigils/) - though YSH only knows the two sigils `$` and `@` (and they use the same namespace). `@bad_log_files` is roughly comparable to `"${bad_log_files[@]}"` and adds all list elements as arguments (without word splitting!).

This also shows the backwards-compatibility of YSH and the focus on only changing/extending where necessary.
It's still perfectly valid to use a POSIX test like `if test -f myfile`.

The two `...` in the following block allow us to define a single command spanning across multiple lines.
This spares us the ` \\ ` at the end of each line and allows us to put comments in between:

```bash
  ... cat ./PROMPT_TEMPLATE.md
      # Ignore all lines from abuild removing packages at the end
    <(... grep -v "Purging" $file | head -n -1
      # the error should be in the last few lines.
      # chatgpt can't handle too much input!
      | tail -40
      ;)
    | chatgpt
    | tr -d u'\n'
    >> $out_file;
```

So these lines are really just one command-chain, including a subcommand.
It does the exact same thing as this POSIX-compatible one-liner:

```bash
cat ./PROMPT_TEMPLATE.md <(grep -v "Purging" $file | head -n -1 | tail -40) | chatgpt | tr -d u'\n' >> $out_file;

```

The command chain does the following:
- Concatenate the `PROMPT_TEMPLATE.md` and the output of the following subshell:
   - Remove all lines saying `Purging` from the log-file (The logs usually ended with many `Purging BUILD-DEPENDENCY` lines)
   - Remove everything except for the last 40 lines
- Feed it to ChatGPT
- Remove any newlines in the ChatGPT output (which shouldnâ€™t be more than 5 words anyway!!1! ðŸ˜‰)

Well that was... interesting!
Firstly, ChatGPT did **NOT** always listen to my instruction to use *only 5 words*. My assumption is that the LLM lost the context in the few cases where it produced huge amounts of text. At least some of these logs contained lines with over 1000 characters.
I did also notice a log message that gives advice on debugging possible errors. This could cause a prompt injection and thus vastly different LLM output. But upon verification I couldn't find the advice in any of the extreme outliers.

Looking at the categories themselves it's obvious that there is something messed up with the compiler in my build environment.

## Conclusion

The script worked and categorized my logs... Definitely not as good as manual categorization would've been, though.
So, am I satisfied with the result?

Yes and no.

- Yes because even though I might've realized that something is wrong after manually categorizing only a few logs, I can be weirdly stubborn. I could've gone on for way too long before finally deciding that I should fix my environment before looking at more logs. It was much more fun to spend a similar amount of time asking the AI instead.
- No because the LLM output was... well... very bad. I'm not sure if I could've gotten a much better result with a few improvements like feeding the already existing categories, or tweaking the Prompt.

For the lacking quality - and the fact that on the second try there were only 215 failed packages to categorize - I switched to manual categorization after fixing the build environment. It took me around the same time to categorize 215 logs (with a script) as writing the script to generate the categories. So if I ever have to categorize >1000 logs, I'll definitely try AI again.

The next blog post will focus on what efforts we took to create a reliable and fast build environment - which gives us good and simple metrics. Because Andy Chu secured an NLNet fund to support us in fixing all these issues, bringing Oils closer to being a real alternative Unix System shell.
